# TradeGuard Context Handoff Manual

## Purpose

This is the fastest possible context handoff for a new engineer or operator.
If someone joins the project today, this document should give them enough context to continue without re-interviewing the founder.

## Project Summary

TradeGuard is a B2B pre-trade risk, evidence preservation, and dispute activation tool for China-based cross-border sellers.

The first MVP is intentionally narrow:

1. Look up buyer company risk
2. Show a simple credit result
3. Let user upload a contract or chat record
4. Return a notarization certificate
5. Prepare a US legal trigger package

Tariff risk is intentionally deferred to Phase 2.

## Why This Product Exists

The product exists to solve one core pain:

- sellers often enter deals without enough buyer validation
- when disputes happen, they often lack preserved evidence

TradeGuard combines those two moments into one workflow:

- risk check before deal
- evidence preservation before dispute
- legal activation when dispute becomes real

## What the Founder Wants

The founder does not want a broad platform first.
The founder wants one chain to work:

1. input company
2. get risk result
3. upload contract
4. receive certificate
5. anchor evidence
6. generate legal package

This is the primary success metric for engineering at the current stage.

## Tech Stack Decision

### Mobile

- Flutter

Why:

- founder already prefers Flutter direction
- single codebase is enough for MVP speed

### Backend

- NestJS

Why:

- clean modules for credit, evidence, notifications
- easy API growth after MVP

### Database

- Supabase

Why:

- fast setup
- simple hosted Postgres
- enough for logs, evidence metadata, and certificate records

### Automation

- n8n

Why:

- founder explicitly wants automation as a middle layer
- notarization and notification fit workflow orchestration well

## Important Product Decisions Already Made

### Decision 1

Use rule-based credit grading first, not AI-generated grading.

Reason:

- easier to debug
- more deterministic
- less risky for MVP trust

### Decision 2

Use AI only later for text summaries, not core decisions.

Reason:

- risk tools must remain explainable

### Decision 3

Tariff logic is Phase 2, not now.

Reason:

- too much scope for first shipping chain
- separate data model and compliance complexity

### Decision 4

Evidence preservation is not the end-state product.

Reason:

- preserved evidence alone does not help users start action fast enough
- the product should bridge preserved evidence into a US legal workflow
